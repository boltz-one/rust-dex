## Component Interaction Flow

### Application Startup

1. **OS Initialization** (`crates/app/src/main.rs`)
   - App binary linked to `gpui_platform` facade
   - Calls `gpui_platform::application()` вҶ’ routes based on `#[cfg(target_os = "...")]`

2. **Platform Backend Selection**
   - macOS: `gpui_macos::platform::Platform` initialized with Metal renderer
   - Linux: `gpui_linux::platform::Platform` with Wayland/X11 dispatcher
   - Windows: `gpui_windows::platform::Platform` with DirectX setup

3. **Theme Initialization**
   - `theme::init(LoadThemes::JustBase, cx)` called
   - Built-in fallback themes (One Dark) loaded into `AppContext`
   - `BaseThemeSettingsProvider` supplies font/density settings

4. **Window Creation**
   - `App::open_window()` called with `HelloWorldApp` entity
   - Platform backend creates native window (Cocoa NSWindow / X11/Wayland window / Win32 HWND)
   - GPUI frame loop started

5. **Render Loop**
   - Platform event dispatcher feeds input/lifecycle events to GPUI
   - GPUI calls `HelloWorldApp::render()` on each frame
   - Flexbox layout calculated
   - GPU command buffers generated by platform renderer
   - Display compositor swaps buffers


## Platform Backend Abstraction

### Interface Definition

**File**: `crates/gpui/src/platform.rs`

Defines traits:
- `Platform` вҖ” windowing, event dispatch, display info
- `Window` вҖ” frame rate, input context, vsync
- `Dispatcher` вҖ” thread pool for async tasks
- `Clipboard` вҖ” copy/paste integration

Each platform backend implements these traits.

### Platform Facade Implementation

**File**: `crates/gpui_platform/src/gpui_platform.rs`

```rust
pub fn application() -> gpui::Application {
    #[cfg(target_os = "macos")]
    {
        gpui_macos::platform::Platform::new().into_app()
    }
    #[cfg(target_os = "linux")]
    {
        gpui_linux::platform::Platform::new().into_app()
    }
    #[cfg(target_os = "windows")]
    {
        gpui_windows::platform::Platform::new().into_app()
    }
}
```

**Single decision point** вҖ” `#[cfg]` gates centralized here only. App code never branches.

### macOS-Specific Details

**File**: `crates/gpui_macos/src/platform.rs`

- **Windowing**: Cocoa `NSApplication`, `NSWindow` managed
- **Rendering**: Metal rendering via `MTLCommandQueue`
- **Text**: Core Text font loader + Text Kit shaping
- **Clipboard**: NSPasteboard integration
- **Display**: DisplayLink for vsync (60 fps target)
- **Shaders**: Metal shading language in `src/shaders.metal`

### Linux-Specific Details

**File**: `crates/gpui_linux/src/linux/platform.rs`

Two display backends:
1. **Wayland** (`src/linux/wayland/`)
   - Compositing via wayland-client protocol
   - Input via Wayland seat/pointer/keyboard
   - XDG surface management
   - XDG Shell (app windows)

2. **X11** (`src/linux/x11/`)
   - X11 connection via xcb library
   - ICCCM/EWMH standards
   - XIM for text input
   - Fallback to X11 if Wayland unavailable

Rendering: wgpu backend (shared in `crates/gpui_wgpu`)

### Windows-Specific Details

**File**: `crates/gpui_windows/src/platform.rs`

- **Windowing**: Win32 CreateWindow / DefWindowProc message loop
- **Rendering**: DirectX 12 command lists
- **Text**: DirectWrite API for font enumeration & shaping
- **Clipboard**: Win32 clipboard API
- **Input**: WM_KEYDOWN, WM_LBUTTONDOWN, etc. via message queue

## Theme System Architecture

### Theme Initialization

**File**: `crates/theme/src/lib.rs`

1. **LoadThemes Enum**
   ```rust
   pub enum LoadThemes {
       JustBase,            // Built-in fallback only (template default)
       All(Box<dyn AssetSource>), // Load from files + fallback
   }
   ```

2. **Builtin Themes**
   - One Dark (default)
   - Additional built-in palettes (no file I/O needed)

3. **Theme Application**
   - `ThemeSettingsProvider` trait supplies font, font size, density
   - `App` calls provider methods during render
   - Colors pulled from `SystemColors` or `UserTheme`

### Theme Customization Path

To extend:
1. Add new color palette to `crates/theme/src/colors.rs`
2. Load via `theme::init(LoadThemes::JustBase, cx)`
3. To bundle theme files:
   - Create `assets/themes/` folder with JSON files
   - Re-implement `AssetSource` trait (or use `rust-embed`)
   - Switch to `LoadThemes::All(Box::new(your_assets))`

## Font System Architecture

### Font Discovery & Loading

**File**: `crates/font_kit/` (vendored)

Platform-specific loaders:
- **macOS**: Core Text (`src/loaders/core_text.rs`)
  - System font lookup: `/Library/Fonts`, `/System/Library/Fonts`
  - Font matching via CTFont APIs
  
- **Linux**: FreeType + fontconfig (`src/loaders/freetype.rs`, `src/sources/fontconfig.rs`)
  - Query fontconfig for available fonts
  - Load via FreeType

- **Windows**: DirectWrite (`src/loaders/directwrite.rs`)
  - System font enumeration via IDWriteFactory
  - Fallback to GDI for basic fonts

### Font System in GPUI

**File**: `crates/gpui/src/text_system.rs`

1. **Font Request** вҶ’ app specifies `Font { family: "Menlo", ..}`
2. **Platform Lookup** вҶ’ font_kit finds best match on system
3. **Font Metrics** вҶ’ dimensions, ascent, descent calculated
4. **Fallback Chain** вҶ’ missing glyphs fall through to system defaults
5. **Rasterization** вҶ’ glyphs rendered to texture atlas (platform-specific)

No bundled fonts; all system fonts.

## Text Rendering Pipeline

### Shared Text System Components

**File**: `crates/gpui/src/text_system/`

1. **Line Wrapping** (`line_wrapper.rs`)
   - Break text at word boundaries
   - Respects layout width constraints

2. **Line Layout** (`line_layout.rs`)
   - Glyph positioning within line
   - Kerning & ligatures (via platform shaper)

3. **Font Features** (`font_features.rs`)
   - Enable/disable OpenType features

### Platform-Specific Text Rendering

- **macOS** (`crates/gpui_macos/src/text_system.rs`)
  - Core Text shaping
  - Core Graphics rasterization

- **Linux** (`crates/gpui_wgpu/src/cosmic_text_system.rs`)
  - cosmic-text library for shaping & rasterization
  - Shared across Linux & Windows

- **Windows** (`crates/gpui_windows/src/text_system.rs`)
  - DirectWrite shaping
  - Rasterization to texture atlas

## Asset Cache

**File**: `crates/gpui/src/asset_cache.rs`

Caches:
- Rendered text glyphs (texture atlas)
- Decoded images
- SVG rasterization

Lifecycle:
1. Asset requested via image/text API
2. Cache lookup (miss = load + rasterize)
3. Texture uploaded to GPU
4. Reference kept until frame cleanup
5. LRU eviction if memory pressure

No file system: images come from memory buffers or embedded data.

## Input System

**File**: `crates/gpui/src/input.rs`, `crates/gpui/src/key_dispatch.rs`

### Event Types

- **Pointer Events**: `PointerMoved`, `PointerPressed`, `PointerReleased`, `PointerScrolled`
- **Keyboard Events**: `KeyDown`, `KeyUp`, `ModifiersChanged`
- **Lifecycle**: `WindowActivated`, `WindowResized`, `WindowCloseRequested`

### Key Dispatch

1. Platform generates key event (e.g., Core Text IME input on macOS)
2. GPUI converts to `KeyDown` with keycode + modifiers
3. Context router finds focused view
4. Action dispatcher matches key binding
5. Action handler called (or event propagates to parent)

## Animation System

**File**: `crates/gpui/src/elements/animation.rs`

- Value interpolation over time
- Easing functions (ease-in, ease-out, linear, etc.)
- Per-element animations
- GPU acceleration via shader

## Dependency Injection (Context System)

**File**: `crates/gpui/src/app/context.rs`

GPUI uses context as DI container:
- `AppContext` вҖ” global app state (themes, globals)
- `WindowContext` вҖ” window-specific state
- `ViewContext<T>` вҖ” entity state

Accessed via `cx` parameter in `render()`, event handlers, etc.:
```rust
fn render(&mut self, _: &mut Window, cx: &mut Context<Self>) -> impl IntoElement {
    let theme = cx.theme();           // Read current theme
    let globals = cx.global::<State>(); // Read global state
    // ...
}
```

## Build System Integration

### Runtime Shaders Feature

**Feature**: `gpui_platform/runtime_shaders`

- On macOS: avoids requiring Metal Xcode toolchain
- Shaders compiled at runtime (slower build, faster iteration)
- Default for `make dev`

### Platform Selection During Build

- `cargo build -p app --target=x86_64-apple-darwin` вҶ’ links `gpui_macos`
- `cargo build -p app --target=x86_64-unknown-linux-gnu` вҶ’ links `gpui_linux`
- `cargo build -p app --target=x86_64-pc-windows-msvc` вҶ’ links `gpui_windows`

No conditional feature; target triple alone selects platform.

## Extensibility Points

### Add a Custom Component

1. Create module in `crates/ui/src/components/my_component.rs`
2. Implement `Render` trait
3. Export from `crates/ui/src/lib.rs`
4. Use in app

### Add a Custom Theme

1. Extend `crates/theme/src/colors.rs` with new palette
2. Call `theme::init(LoadThemes::JustBase, cx)` (uses new color)
3. Or implement `AssetSource` + switch to `LoadThemes::All(...)`

### Add Platform-Specific Behavior

1. Add method to `crates/gpui/src/platform.rs` trait
2. Implement in `gpui_macos`, `gpui_linux`, `gpui_windows`
3. Call via `cx.platform()` (if context accessible)

### Add Persistence

1. Create `crates/db/` with SQLite layer
2. Add to workspace members
3. Call from app to load/save state on startup/shutdown
4. No changes to GPUI core needed

## Security & Safety

### Memory Safety

- All Rust code; no unsafe except where required (FFI, OS APIs)
- Platform backends (Metal, DirectX, X11 FFI) isolated and audited
- No buffer overflows possible; bounds checked

### Privilege Isolation

- No elevated privileges required (app runs as user)
- File access via system APIs (XDG portal on Linux for sandboxing)
- No network by default (optional via `http_client` crate)

### Asset Handling

- No untrusted file loading (by default)
- Theme/icon loading from built-in data (no file I/O)
- Images can be loaded from user-provided buffers (app responsibility)

## Performance Characteristics

### Expected Frame Time

- **Simple layout** (hello world): <1ms GPUI + <5ms GPU rendering = 60 fps easily
- **Complex layout** (100s of elements): GPUI layout ~5-10ms, GPU ~10-20ms (depends on GPU)
- **Text rendering**: First frame ~50ms (atlas population), subsequent <1ms

### Memory Footprint

- Base app: ~50-100 MB RSS (GPUI runtime + frameworks)
- Per-window: ~10-20 MB for buffers, texture atlases
- Scalable; no bloat from unused features
