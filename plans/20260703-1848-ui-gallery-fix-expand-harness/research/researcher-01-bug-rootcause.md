# ui_gallery interaction bugs — root cause research

Method: read gallery_app.rs, main.rs, pages/*.rs, and crates/ui component sources (text_input, tab.rs/tab_bar.rs, multi_select.rs, combobox.rs, search_input.rs, segmented_control.rs). No code changed.

## Bug matrix

### 1. Scroll không hoạt động trên các page
- **Root cause**: no page container anywhere in `examples/ui_gallery/src/` uses a scroll pattern. `grep -rn "overflow_y_scroll|overflow_scroll|Scrollbar|scroll_handle|UniformList|list("` over `examples/ui_gallery/src/` → 0 hits. The content wrapper is a plain `v_flex` with no id/overflow.
- **Evidence**: `examples/ui_gallery/src/gallery_app.rs:141` — `.child(v_flex().flex_1().p_6().gap_8().child(content))` has no `.id(..)`, `.overflow_y_scroll()`, or `ScrollHandle`.
- **Correct pattern exists in repo** (not used by gallery): `crates/ui/src/components/scrollbar.rs`, `data_table.rs`, `modal.rs`, `drawer.rs`, `tab_bar.rs`, `context_menu.rs`, `navigable.rs`, `indent_guides.rs`, `callout.rs` and `crates/gpui/src/elements/uniform_list.rs` all use `ScrollHandle`/`Scrollbar::`/`overflow_y_scroll`/`uniform_list`. GPUI's div needs a stable `id()` + `.overflow_y_scroll()` (optionally paired with `Scrollbar` for a visible thumb) to become scrollable; without an `id`, scroll offset can't be tracked across frames either.
- **Fix direction**: wrap the page-content container in `gallery_app.rs` (or per-page) with `.id("gallery-content").overflow_y_scroll()`, mirroring `crates/ui/src/components/scrollbar.rs`'s usage.
- **LOẠI**: gallery-wiring (examples/ui_gallery only; component-level scroll primitives already exist in crates/ui).

### 2. Click đổi Tabs không có event / nội dung không đổi
- `TabBar`/`Tab` is **not referenced anywhere** in `examples/ui_gallery/src/` (`grep -rn "Tabs|TabBar|tab_bar|Tab::"` → 0 hits) — it isn't shown in the gallery at all, so the reported "Tabs" is almost certainly **SegmentedControl** (shown on Forms page, visually tab-like).
- **Root cause confirmed**: `crates/ui/src/components/forms.rs`(gallery)/`segmented_control.rs:116-124` — `SegmentedControl::preview()` builds `SegmentedControl::new(...).active(1)` and never calls `.on_change(...)`. The click handler exists in the component itself (`segmented_control.rs:92-98`: `if !disabled { cell.on_click(move |_,window,cx| if let Some(handler)=&on_change { handler(i,window,cx) }) }`) but fires into `on_change = None` in preview mode → click registers, no-op executes, `.active(1)` stays hardcoded forever.
- Separately, if the real target IS `TabBar`: `crates/ui/src/components/tab_bar.rs:201-240`'s `preview()` builds `Tab::new(..).toggle_state(true)` with **no `.on_click` at all** wired (grep for `on_click|active(` in `tab_bar.rs` → 0 hits) — `Tab` does implement `InteractiveElement`/`StatefulInteractiveElement` (tab.rs:61,67) so it CAN take `.on_click`, but the static preview never attaches one. Purely decorative mockup.
- **Fix direction**: gallery needs to hold an `Entity`/local state for "active segment/tab" and wire `.on_change`/`.on_click` to mutate it + `cx.notify()`, instead of embedding the component's static `::preview()`.
- **LOẠI**: gallery-wiring (forms.rs uses decorative `::preview()` instead of a real stateful demo); the underlying component's `on_click`/`on_change` plumbing is fine.

### 3. MultiSelect chọn nhiều không cập nhật
- **Root cause**: `examples/ui_gallery/src/pages/forms.rs:53` calls `multi_select_preview(window, cx)` inside `render_forms`, a method on `GalleryApp` invoked fresh on **every** `GalleryApp::render()`. `crates/ui/src/components/multi_select.rs:151-157` — `multi_select_preview()` does `cx.new(|_| MultiSelect::new(options).selected_indices([0, 2]))` **inside the free preview function itself**, not stored in `GalleryApp`'s struct. Every parent re-render (sidebar switch, theme toggle, any `cx.notify()` upstream) creates a **brand-new `Entity<MultiSelect>`**, discarding the previous one and resetting selection back to the hardcoded `[0, 2]`.
- Click handlers inside `MultiSelect` itself are correctly wired (`multi_select.rs:86,121-137` use `cx.listener` mutating `self.selected` + presumably `cx.notify()`), so a click updates state for that one entity instance — but the instance is thrown away on the next parent render, making it look like "selection doesn't update."
- **Fix direction**: store the `Entity<MultiSelect>` in `GalleryApp` (like `text_input`/`textarea`/`select` already are, `gallery_app.rs:57-59,69-75`) and pass the persisted handle into `render_forms` instead of calling `multi_select_preview()`.
- **LOẠI**: gallery-wiring. Same defect class applies to `combobox_preview` (`combobox.rs:137-141`, `cx.new` inside preview fn) and `search_input_preview` (`search_input.rs:60-64`, same pattern) — both are stateful (own an inner `Entity<TextInput>`) yet recreated every render, so any typed search text / open dropdown state is silently lost on the next unrelated re-render.

### 4. TextInput/input KHÔNG gõ được text
- **Root cause (component-bug)**: `crates/ui/src/components/text_input.rs` — `on_key_down` (line 64-83) is correctly wired via `.on_key_down(cx.listener(Self::on_key_down))` (line 113) and mutates `self.content` + `cx.notify()`, and the div does `.track_focus(&self.focus_handle)` (line 112). **But nothing ever calls `window.focus(&self.focus_handle)`** — no `.on_click`/`.on_mouse_down` in the whole render fn (lines 111-132 read in full, confirmed no focus-grab call). `track_focus` only registers the div as focusable/participates in the focus tree; it does not auto-focus on click in GPUI (Zed-derived convention requires an explicit `window.focus(...)` call in a mouse-down handler — see `[unverified]`, not confirmed against GPUI internals here, listed as open question).
- Confirms: `GalleryApp` also never calls `window.focus(...)` anywhere (`grep -n "focus|Focus" gallery_app.rs main.rs` → 0 hits), so there's no fallback initial focus either. Net effect: the input's `FocusHandle` is never focused, key events never dispatch to `on_key_down`, so typing does nothing regardless of correct key-handling logic.
- **Fix direction**: add `.on_mouse_down(MouseButton::Left, cx.listener(|this, _, window, cx| window.focus(&this.focus_handle)))` (or `.on_click`) to `text_input.rs`'s render div.
- **LOẠI**: component-bug (`crates/ui/src/components/text_input.rs`), not gallery-specific — every consumer of `TextInput` is affected, including `Combobox`/`SearchInput` which embed it.

### 5. Other bugs found while reading
- **Entity-recreated-in-render antipattern also affects**: `Combobox` (`combobox.rs:137-141`) and `SearchInput` (`search_input.rs:60-64`) previews — same "new Entity every render" issue as MultiSelect (#3). `InputGroup::preview`, `FormField::preview`, `ActionPanel::preview`, `FileInput::preview` (forms.rs:45,59-61) were not individually source-checked — flagged for follow-up.
- **SegmentedControl's `disabled` variant** (`segmented_control.rs:126-128`) is intentionally non-interactive — not a bug, expected.
- Static-page functions (`pages::elements::render`, `pages::navigation::render`, `pages::data::render`, `pages::feedback::render`, `pages::layout::render`) are plain `fn(window, cx) -> AnyElement` (not methods on `GalleryApp`), so **any** `::preview()` in those pages that is genuinely stateful (not yet audited: data.rs/feedback.rs/overlays.rs/layout.rs contents were not read this pass) likely has the same recreate-per-render risk as #3 wherever `cx.new(...)` appears inside a `*_preview` free function. Recommend grepping `cx.new(` inside all `*_preview(` fns across `crates/ui/src/components/*.rs` before fixing.
- `gallery_app.rs:100-108` — page switch dispatch (`match current { ... }`) and Forms/Overlays routed through `self.render_forms`/`self.render_overlays` (methods, correctly `&self`) while others are free fns — inconsistent but not itself a bug; just means Forms/Overlays are the only pages that COULD hold gallery-level entity state today, and even they don't for MultiSelect/Combobox/SearchInput (see #3).

## GPUI patterns confirmed correct in this repo (for fix reference)
- **Stateful widget = persisted `Entity<T>` owned by parent struct**, created once in `::new()`/constructor, passed by `.clone()` into render — see `gallery_app.rs:57-59` (`text_input`, `textarea`, `select`) — this is the pattern MultiSelect/Combobox/SearchInput/Tabs/SegmentedControl demos must follow instead of static `::preview()`.
- **Scroll**: needs `.id(..)` + `.overflow_y_scroll()` (see `crates/ui/src/components/scrollbar.rs`, `data_table.rs`, `tab_bar.rs` usage) — gallery currently has none.
- **Focus-to-type**: `FocusHandle` + `.track_focus()` + `.on_key_down()` is necessary but not sufficient; an explicit focus-grab (`window.focus(&handle)`) on click/mouse-down is required and missing in `text_input.rs`.

## Open questions
- Does GPUI's `.track_focus()` auto-grant focus on mouse-down, or is an explicit `window.focus()` call strictly required? Not verified against `crates/gpui` focus-dispatch internals — if auto-focus exists, bug #4's root cause would shift elsewhere (e.g. modifier-key gating at `text_input.rs:67`, or event not reaching the div due to z-order/hit-test). Recommend tracing `crates/gpui/src/window.rs` focus dispatch before implementing the fix.
- `pages/data.rs`, `pages/feedback.rs`, `pages/overlays.rs`, `pages/layout.rs` were not read this pass — unverified whether they contain additional recreate-per-render or unwired-preview instances beyond Forms.
- `InputGroup::preview`, `FormField::preview`, `ActionPanel::preview`, `FileInput::preview` (forms.rs) not source-checked for the same antipattern.
- Whether `cx.notify()` inside a child `Entity`'s own listener (e.g. `MultiSelect`'s click handler) actually forces the *parent* (`GalleryApp`) to re-render and thus re-execute `multi_select_preview()` — this determines exact trigger frequency of the reset in bug #3, but doesn't change the fix (persist the Entity in `GalleryApp`).
